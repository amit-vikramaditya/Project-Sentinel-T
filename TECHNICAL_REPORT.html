Abstract

The Controller Area Network (CAN) protocol remains the dominant in-vehicle communication standard across the global automotive industry. Its original 1986 design, however, assumed a trusted, physically sealed bus—an assumption that modern connected vehicles violate. CAN lacks authentication, encryption, and sender identification at the protocol level.

This report presents Project Sentinel-T, a Physical-Layer Intrusion Detection System (IDS) that authenticates Electronic Control Units (ECUs) not by their data, but by their hardware identity. Sentinel-T exploits a fundamental asymmetry: the crystal oscillator inside every ECU exhibits a unique, thermally-driven clock drift that is physically impossible for a software attacker to replicate. By intercepting CAN frames at the Linux Kernel level using SO_TIMESTAMP socket options and modeling each sender's clock behavior with a Phase-Velocity Kalman Filter, Sentinel-T can distinguish a legitimate ECU from a software-injected spoofed message in real time.

The system was deployed and validated on a Microsoft Azure Cloud Virtual Machine running Ubuntu 24.04 with linux-modules-extra kernel extensions and a vcan0 virtual CAN interface. The Kalman Filter was tuned with Process Noise Q = 1×10⁻¹² and Measurement Noise R = 1×10⁻¹⁰. Under live testing, a valid ECU (cangen) produced an average residual error of 0.53 µs, while a software attacker (Bash cansend loop) exhibited errors of 350 µs – 800 µs. The achieved Signal-to-Noise Ratio (SNR) was 1.88×, confirming viable detection at a threshold of 200 µs with per-packet latency < 0.04 ms.

Introduction

1.1 Background

The Controller Area Network (CAN) protocol, standardized as ISO 11898, was designed by Robert Bosch GmbH in 1986 to enable robust, low-cost communication between microcontrollers inside a vehicle without a host computer. It has since become the de facto standard for in-vehicle networking, connecting dozens to over a hundred Electronic Control Units (ECUs) in a modern automobile—from engine management and anti-lock braking systems (ABS) to airbag controllers and advanced driver-assistance systems (ADAS).

CAN operates as a multi-master, message-broadcast serial bus. Every message transmitted on the bus is visible to every connected node. Messages are identified not by sender address but by a message identifier (CAN ID), which also serves as the arbitration priority. There is no source address field, no authentication header, and no encryption layer.

The modern attack surface has expanded dramatically through:

  • OBD-II diagnostic ports, physically accessible under the dashboard.
  • Telematics Control Units (TCUs) with cellular connectivity.
  • Infotainment systems with Wi-Fi, Bluetooth, and USB interfaces.
  • V2X (Vehicle-to-Everything) communication modules.
  • Aftermarket dongles plugged into diagnostic ports.

1.2 Problem Statement

The fundamental problem is that CAN has no mechanism to verify who sent a message. An attacker who gains access to the bus can inject arbitrary frames indistinguishable from legitimate traffic at the protocol level.

The obvious countermeasure—cryptographic Message Authentication Codes (MACs)—faces severe constraints:

  1. Payload limitation. Standard CAN 2.0 carries 8 bytes max. A truncated HMAC consumes half or more of this payload.
  2. Legacy hardware. Hundreds of millions of vehicles use 8/16-bit microcontrollers lacking cryptographic accelerators.
  3. Real-time constraints. Crypto latency risks violating hard real-time deadlines in safety-critical control loops (1–10 ms cycles).
  4. Key management. Distributing and rotating keys across 70–100 ECUs from multiple Tier-1 suppliers is a supply-chain problem of significant complexity.

These constraints motivate physical-layer authentication: analyzing the physical timing characteristics of how a message arrives rather than adding information to what it contains.

2.2 The Kernel Tap Mechanism

The precision of Sentinel-T depends entirely on timestamp quality. There are two approaches:

❌ User-Space Timestamping (time.time())

The naive approach: call time.time() after receiving a frame. This timestamp reflects when the Python interpreter executes the syscall—not when the frame arrived. The delta is dominated by OS scheduling latency (50–500 µs), interrupt coalescing, and GIL overhead. This jitter is the same magnitude as the drift signal, making it useless.

✅ Kernel-Space Timestamping (SO_TIMESTAMP)

The SO_TIMESTAMP socket option records the timestamp at frame delivery to the socket buffer—inside the kernel, before user-space scheduling. Retrieved via recvmsg() ancillary data:

# Enable kernel-level timestamping
SO_TIMESTAMP = 29
self.sock.setsockopt(SOL_SOCKET, SO_TIMESTAMP, 1)

# Receive frame + kernel timestamp
msg, ancdata, flags, addr = self.sock.recvmsg(16, cmsg_capacity)

# Extract struct timeval from ancillary data
for cmsg_level, cmsg_type, cmsg_data in ancdata:
    if cmsg_level == SOL_SOCKET and cmsg_type == SCM_TIMESTAMP:
        seconds, microseconds = struct.unpack("qq", cmsg_data)
        kernel_timestamp = seconds + (microseconds / 1000000.0)

2.3 The Chronomorphic Engine

The term "chronomorphic" (from Greek chronos + morphe) refers to the principle that the shape of time—the pattern of inter-arrival intervals—encodes the physical identity of the sender.

Thermal Drift (Real ECU) — PHYSICAL

Every ECU's clock is driven by a quartz crystal oscillator with a temperature-dependent resonant frequency. This drift is correlated in time, unique per device, and physically bound. Modeled as sinusoidal thermal drift + Ornstein-Uhlenbeck mean-reverting process:

thermal_drift = np.sin(np.linspace(0, 4, N)) * 0.00002
current_jitter += -theta * current_jitter + sigma * np.random.normal()

Software Jitter (Attacker) — ANOMALY

Software attackers rely on OS timing (sleep(), usleep()). This produces Gaussian, uncorrelated, memoryless noise—a fundamentally different statistical fingerprint that the Kalman Filter is designed to reject.

Methodology

3.1 State Space Modeling

The core engine is a two-dimensional Phase-Velocity (PV) Kalman Filter. The state vector:

xk = [ φk  ,  φ̇k ]T

φk = phase offset (accumulated timing deviation, seconds)
φ̇k = frequency drift (rate of change, seconds/interval)

State Transition Matrix (F):

F = │ 1  1 │    →   x_pred = F × x
    │ 0  1 │

Observation Matrix: H = [1   0] — only phase offset is directly observed.

Kalman Recursion (per packet):

  1. Predict: x_pred = F @ x , P_pred = F @ P @ Fᵀ + Q
  2. Measure: residual = z - H @ x_pred , where z = observed_interval - base_interval
  3. Update: K = P_pred @ Hᵀ @ S⁻¹ , x = x_pred + K × residual

The residual is the critical output: it measures how surprised the filter is. For a legitimate ECU, residuals converge to near-zero. For an attacker, residuals remain persistently elevated.

Q (Process Noise) 1×10⁻¹² "Physical drift rate changes negligibly between 10 ms measurements. Trust the model strongly."
R (Measurement Noise) 1×10⁻¹⁰ "Individual measurements are moderately noisy (~10 µs σ). Don't overreact to any single observation."

4.2 Software Modules

4.2.1 can_receiver.py — The Kernel Tap

The CANReceiver class opens a PF_CAN / SOCK_RAW / CAN_RAW socket, enables SO_TIMESTAMP (option 29), binds to vcan0, and uses recvmsg() to receive both the 16-byte CAN frame and the struct timeval kernel timestamp from ancillary cmsg data. Parses CAN frame format <IB3x8s> and handles EFF/SFF ID masking.

4.2.2 drift_tracker.py — The Kalman Engine

The DriftTracker class implements the 2D PV Kalman Filter. Provides update(interval) for simulation, update_from_can_socket(timestamp) for live kernel timestamps (computes intervals by differencing), and process_stream(intervals) for batch analysis. Initial state: [0, 0]ᵀ with P = 0.1 × I₂.

4.2.3 live_sentinel.py — The Real-Time Monitor

The operational entry point. Creates per-CAN-ID DriftTracker instances dynamically. For each frame: retrieves kernel timestamp → updates tracker → computes residual in µs → applies WARMUP / PHYSICAL / ANOMALY classification → prints color-coded dashboard. Runs until Ctrl+C.

05

Conclusion & Future Work

5.1 Conclusion

  1. Kernel timestamping is essential. SO_TIMESTAMP reduces measurement noise by 1–2 orders of magnitude vs. time.time().
  2. The Kalman Filter is effective. A simple 2-state PV filter (Q=1×10⁻¹², R=1×10⁻¹⁰) separates correlated thermal drift from uncorrelated scheduling jitter.
  3. The detection boundary is clear. 200 µs threshold cleanly separates 0.53 µs (ECU) from 350–800 µs (attacker) with zero false positives/negatives.
  4. The system is fast. < 0.04 ms per-packet latency leaves > 99.6% of the 10 ms cycle for other processing.
  5. Cloud validation works for Phase 1. Azure VM + vcan0 provided a controlled, repeatable testbed.

The achieved SNR of 1.88× confirms the fundamental hypothesis: physical clock drift contains enough structured information to be distinguishable from software-generated timing.

5.2 Future Work

Phase 2: Physical Hardware Deployment

Deploy on Arduino-based CAN transceiver (MCP2515 + SN65HVD230) connected to a Raspberry Pi. Validate whether 1.88× SNR holds with real crystal oscillator drift, physical bus propagation delays, and hardware interrupt-driven reception.

Phase 3: Mamba SSM Integration

Replace the Kalman Filter with a Mamba State-Space Model (selective SSM) to learn non-linear clock drift patterns from data without hand-tuned Q/R. Potential to handle non-linear thermal responses and aging effects automatically.

Phase 4: Adversarial Robustness

Test against adaptive attackers who deliberately introduce correlated timing to mimic physical drift. Establish theoretical limits of timing-based authentication.
